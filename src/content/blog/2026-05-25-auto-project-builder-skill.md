처음 설계 문서를 열었을 때 예상했던 분량은 200줄이었다.

결과물은 1,400줄짜리 SKILL.md가 됐다. 그 안에는 Phase -0.5부터 Phase 3까지 9단계 파이프라인, OMC/ECC/none 세 가지 모드별 분기, 체크포인트 재개 로직, 자동 QA 루프, 고도화 제안 HTML 생성, 누적형 overview.html 업데이트 알고리즘까지 들어갔다.

`auto-project-builder`는 트렌드 조사부터 GitHub push까지 사람의 개입 없이 실제로 동작하는 프로젝트를 완성하는 스킬이다. 아이디어를 고르고, 경쟁 서비스를 분석하고, PRD를 쓰고, Sprint 계획을 세우고, 코드를 짜고, QA를 돌리고, README를 생성하고, GitHub에 올린다. 이 모든 단계를 자율로.

처음에는 훨씬 단순했다. "Claude Code로 Next.js 앱 하나 자동으로 만들어 주는 스킬"이 출발점이었다. 그런데 구체적으로 설계하면서 자꾸 빈틈이 보였다. 아이디어는 어디서 나오나? 아이디어 품질은 어떻게 보장하나? 빌드가 실패하면? 이전에 실행하다 멈추면? 여러 개를 만들 때는? 각 질문에 답하다 보니 파이프라인이 하나씩 더 생겼다.

야망이 현실을 이겼다. 그래서 지금 이 스킬은 쓰기가 좀 두렵다. 잘 돌아갈 때는 마술 같은데, 엣지 케이스에 걸리면 어디서 멈췄는지 추적하는 것도 일이다.

## 아이디어 생성을 따로 스킬로 분리한 이유

스킬에서 가장 오래 고민한 부분이 Phase 1이다.

처음 구현은 단순했다. WebSearch로 "2025 trending productivity apps"를 검색하고, 거기서 아이디어 5개를 뽑는 방식이었다. 실제로 돌려보니 결과가 뻔했다. 할 일 앱, 노트 앱, 타이머 앱. 이미 GitHub에 수백 개씩 있는 것들. 트렌드를 따라가면 레드오션으로 직행한다는 걸 실감했다.

그래서 접근을 바꿨다. 트렌드를 쫓는 게 아니라 실제 사용자의 고통을 찾아내는 방식으로. Reddit에서 "I wish there was an app that..." 류의 글을 긁고, Hacker News에서 "Ask HN: Is there a tool for..." 스레드를 수집하고, 앱스토어 1점 리뷰에서 반복되는 불만 패턴을 찾고, GitHub Issues에서 미해결 feature request를 모은다. 이 네 소스를 병렬로 돌리는 것이 멀티소스 페인 마이닝이다.

아이디어 평가는 YC 스타일 15점 스코어카드로 구조화했다.

| 평가 항목 | 배점 | 설명 |
|-----------|------|------|
| 페인 강도 | 1~3점 | 실제 사용자 고통의 증거가 있는가 |
| 시장 규모 | 1~3점 | 충분한 잠재 사용자가 있는가 |
| 독창성 | 1~3점 | 기존 솔루션과 차별점이 있는가 |
| 구현 가능성 | 1~3점 | MVP를 2~3 Sprint 안에 만들 수 있는가 |
| 치명적 결함 | -3~0점 | 사업 모델, 법적 리스크, 기술 장벽 등 |

11점 이상이면 GO, 8~10점이면 CONDITIONAL, 7점 이하면 NO-GO다. 단순히 "좋아 보이는" 아이디어를 뽑는 게 아니라 실패 이유를 먼저 찾는 적대적 평가를 거친다.

이 평가 로직은 처음에 auto-project-builder 안에 직접 넣었다가 별도 스킬(`idea-generator`)로 분리했다. 아이디어 생성 자체가 멀티소스 조사, 3역할 병렬 평가, 스코어카드 판정까지 충분히 복잡한 파이프라인이었고, auto-project-builder의 주 흐름에서 분리하는 게 맞다고 판단했다. 결과적으로 idea-generator는 단독으로도 쓸 수 있는 독립 스킬이 됐다.

planner, architect, critic 세 역할이 병렬로 아이디어를 평가하는 구조도 이 단계에 들어간다. planner는 가치 제안과 타겟 페르소나를 검토하고, architect는 구현 가능성과 Sprint 수를 판단하고, critic은 YC 심사위원처럼 실패 이유를 먼저 나열한다. 한 번 실행할 때 `PROJECT_COUNT × 2`배수의 후보를 만들어 필터링 여유분을 확보하는 것도 나중에 추가한 장치다. 초기엔 딱 PROJECT_COUNT만큼만 만들었더니 평가에서 절반이 NO-GO로 탈락하면 모자라는 문제가 생겼다.

## Phase 번호가 음수가 된 경위

완성된 스킬에는 Phase -0.5, -1, 0, 0-A, 0-B, 1, 1.3, 1.5, 2, 2-1 ~ 2-10, 3, 3-1 ~ 3-3이 있다. 번호를 음수로 시작한 것만 봐도 얼마나 뒤늦게 단계들이 끼어들었는지 알 수 있다.

Phase -0.5는 제일 나중에 추가됐다. 스킬이 oh-my-claudecode 환경에서 실행되는지, everything-claude-code 환경인지, 아무것도 없는 베어 환경인지에 따라 에이전트 체인이 완전히 달라지는데, 이 감지를 Phase 0에서 하면 늦다는 걸 깨달았다. 이미 설정 질문을 다 받고 난 뒤에 "어, 여기는 OMC가 없네" 하면 흐름이 어색해진다. 그래서 -0.5를 만들었다.

USER_IDEA 분기도 초기엔 없었다. "알아서 아이디어 생성" 경로만 있었는데, 사용자가 이미 만들고 싶은 게 있는 경우를 처리해야 했다. 직접 입력 경로가 생기면서 Q3(서비스 유형)와 Q4(프로젝트 개수) 질문이 불필요해졌고, 그 분기가 Phase 1에도, Phase 2에도, Phase 3 보고서 포맷에도 퍼졌다. 한 곳에 추가한 분기가 스킬 전체에 리플이 되는 경험을 했다.

Phase 2를 보면 2-1 PRD 작성부터 2-10 고도화 제안 수집까지 10개 하위 단계가 있다. 처음엔 5단계였다. 순서대로 늘어난 단계들을 보면 스킬 개발의 역사가 보인다. 2-7 완료 알림은 "오래 걸리는 작업인데 완료 소식을 알려줘야 하지 않나"에서 추가됐고, 2-8 체크포인트 저장은 "중간에 실패하면 처음부터 다시 해야 하는 거 아닌가"에서, 2-10 고도화 제안은 "빌드하고 끝이 아니라 다음 할 일도 같이 제안해 주면 어떨까"에서 생겼다.

단계가 많아진 것 자체는 문제가 아니다. 문제는 단계 사이의 상태 전달이 복잡해진다는 점이다. `PLATFORM`, `USER_IDEA`, `TECH_STACK`, `SERVICE_CATEGORIES`, `PROJECT_COUNT`, `STACK_VERSIONS`, `TREND_DATA`, `IDEAS`, `IDEA_SCORES`, `GITHUB_REPOS`, `APPROVED_IDEAS`, `PROJECT_LOG`, `QA_ATTEMPTS`, `RUN_DATE`, `CHECKPOINT_FILE` — 15개 변수가 파이프라인을 흐른다. 변수 추적 테이블을 명세에 따로 넣은 이유가 여기 있다.

## QA 루프와 체크포인트 설계

> "QA를 통과하지 못하면 완성이 아니다." — SKILL.md 첫 번째 원칙

구현하고 나면 빌드 오류 하나쯤은 남는 게 현실이었고, 그걸 자동으로 고치는 루프가 없으면 스킬이 쓸모가 없다.

QA 루프의 구조는 단순하다. `while QA_ATTEMPTS < 3`으로 돌리면서, 실패하면 오류를 파싱해서 debugger 에이전트에 넘기고, 수정 후 재시도한다. 3회 다 실패하면 스코프를 줄인다. Nice-to-have 기능을 제거하고 Must-have만 남겨서 다시 구현한다. 그래도 안 되면 SKIP 마킹하고 다음 프로젝트로 넘어간다.

실제로 테스트해 보니 TypeScript 에러가 제일 자주 걸렸다. 에이전트가 생성한 코드에서 Drizzle 스키마 타입과 실제 쿼리 반환 타입이 맞지 않는 케이스가 많았다. debugger가 대부분 고치는데, 가끔 근본 원인을 찾지 못하고 증상만 억누르는 패치를 하는 경우가 있었다. 2회 재시도 안에 못 고치면 스코프 축소로 넘어가는 이유가 여기 있다.

체크포인트는 `.auto-project-builder-checkpoint.json` 파일 하나로 관리한다. 실행 중에는 존재하고, 전체 완료 시 자동으로 삭제된다. 이 파일이 있으면 스킬 시작 시 "이어서 진행할까요?" 질문이 나온다. 완료된 프로젝트는 체크포인트에 기록되어 있으니 건너뛰고, 남은 것부터 이어서 한다.

완료 시 체크포인트를 삭제하는 설계는 처음엔 찜찜했다. 그런데 완료 보고서(`{RUN_DATE}_report.html`)와 `overview.html`이 모든 실행 이력을 누적하기 때문에 체크포인트 자체는 굳이 남길 필요가 없다. 오히려 완료 후에도 파일이 남아 있으면 다음 실행 시 "이전 작업 이어서 할까요?" 노이즈가 생긴다. 파일의 존재 자체가 "미완료"를 의미하도록 설계하는 게 더 명확하다.

## OMC_MODE 분기: 환경 감지의 교훈

스킬을 처음 설계할 때는 oh-my-claudecode 환경만 가정했다. planner, architect, executor, designer, code-reviewer, security-reviewer, debugger, verifier, writer, git-master 에이전트를 전부 쓰는 풍부한 조합으로.

그런데 everything-claude-code만 있는 환경도 있고, 아무 서드파티 스킬 없이 Claude Code 기본 에이전트만 있는 환경도 있다. 스킬이 어떤 환경에서도 동작해야 한다면 각 환경에 맞는 에이전트 체인을 정의해야 했다.

```
if "oh-my-claudecode:" in available_skills:
  OMC_MODE = "omc"
elif "everything-claude-code:" in available_skills:
  OMC_MODE = "ecc"
else:
  OMC_MODE = "none"
```

감지 로직 자체는 단순하다. 문제는 이 값 하나가 Phase 0부터 Phase 3까지 모든 에이전트 선택을 결정한다는 점이다. `OMC_MODE = "omc"`일 때 planner가 PRD를 쓰고, `OMC_MODE = "ecc"`일 때는 everything-claude-code:planner가 하고, `OMC_MODE = "none"`일 때는 직접 작성한다. 세 분기를 모든 단계에서 유지하다 보니 명세서 분량이 크게 늘어났다.

autopilot과의 통합도 OMC_MODE 분기 문제다. `OMC_MODE = "omc"`일 때만 Phase 2 전체를 autopilot에 위임하는 옵션이 있다. autopilot 자체가 oh-my-claudecode 스킬이기 때문이다. autopilot을 쓰면 Phase 2-1 ~ 2-10을 다 건너뛰는 거라 명세와 실제 실행 경로가 다르다. "이 스킬은 Phase 2를 이렇게 실행한다"고 설명해 놨는데, 실제로는 autopilot이 자기 방식으로 하는 것이다. 이 부분이 지금도 좀 어색하다.

환경 감지를 가장 먼저 하는 것 자체는 맞는 결정이었다. 후속 모든 결정이 환경에 의존하기 때문이다. 하지만 세 환경을 동등하게 지원하는 게 맞는지는 여전히 의문이다. `OMC_MODE = "none"` 경로는 테스트를 적게 했고, 그만큼 품질도 덜 보장된다. 지원 범위를 좁히고 깊이를 높이는 편이 나을 수도 있다.

## 아직 풀리지 않은 것들

실제로 프로젝트 5개를 한 번에 돌렸을 때 토큰 소비가 예상보다 훨씬 많았다. Phase 0의 트렌드 조사, Phase 1의 아이디어 평가, Phase 2의 에이전트 체인이 프로젝트마다 반복되면서 컨텍스트가 폭발적으로 커진다. 10개를 돌리면 중간에 컨텍스트 압축이 일어나고, 그게 이전 프로젝트의 결과를 흐릿하게 만들 수 있다. 체크포인트 재개가 있긴 하지만 재개 직후에도 이전 Phase의 변수(`TREND_DATA`, `APPROVED_IDEAS` 등)를 컨텍스트에 다시 로드하는 명확한 메커니즘이 부족하다.

`overview.html`의 누적 전략도 엣지 케이스가 있다. HTML 마커(`<!-- PROJECT_CARDS_START -->`, `<!-- TIMELINE_START -->`)로 삽입 위치를 찾는 방식인데, 사용자가 직접 HTML을 편집하면 마커가 깨질 수 있다. 그 경우 에이전트가 어디에 삽입해야 하는지 판단하지 못해 기존 내용을 덮어쓸 위험이 있다. append-only 원칙을 명세에 강조했지만, 구현 에이전트가 그 원칙을 지키는지는 실행해봐야 안다.

아이디어 품질과 실제 사용 가능성의 간극도 여전하다. 15점 스코어카드는 기획 가능성을 평가하지, 실제로 만들어진 결과물이 쓸 만한지는 평가하지 않는다. GO 판정을 받고 QA까지 통과한 프로젝트가 실제로는 버그투성이거나, 코어 기능이 형식적으로만 구현되어 있는 경우가 있었다. verifier 에이전트가 이걸 잡아야 하는데, "핵심 기능이 실제로 동작하는지"를 자율 검증하는 능력이 아직 불안정하다. Playwright 기반의 실제 브라우저 테스트를 verifier 체인에 붙이는 게 다음 개선 방향이다.

`ralph` 루프를 QA 실패 2회 이상인 프로젝트에 자동 적용하는 조건을 넣었는데, 실제로 ralph가 acceptance criteria를 모두 통과시키는 데 얼마나 걸리는지, 무한 루프에 빠지는 경우는 없는지가 검증이 덜 됐다. 스킬의 스킬을 자율로 호출하는 중첩 구조라 디버깅이 어렵다.

다음에 고쳐야 할 목록을 정리하면: 컨텍스트 예산 감지 후 자동 배치 분할, overview.html 마커 복구 로직, verifier에 Playwright 연동, ralph 루프 최대 반복 횟수 제한, `OMC_MODE = "none"` 경로 통합 테스트. 기능이 더 필요한 게 아니라 지금 있는 것들이 실제로 잘 돌아가는지 검증하는 데 시간이 더 필요하다.
